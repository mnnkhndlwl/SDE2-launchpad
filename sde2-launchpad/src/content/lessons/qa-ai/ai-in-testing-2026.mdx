AI won't replace testers — but testers who use AI well are replacing those who don't. Treat it as a **force multiplier**, not autopilot.

## Where AI helps today

- **LLM coding assistants** (Copilot, Cursor) — generate and refactor test code, scaffold Page Objects, draft assertions from a natural-language prompt.
- **Self-healing locators** — when a button moves or is renamed, AI-backed tools re-find it by text/context and keep the test passing instead of failing, cutting locator maintenance.
- **AI test generation** — produce test cases / edge cases from requirements, user stories or even a recorded session.
- **Flaky-test detection** — ML flags tests that fail intermittently so you can quarantine and fix them.
- **Visual/AI validation** — detect meaningful UI changes while ignoring trivial pixel noise.

## The AI-native tool landscape

Testim, Mabl, Functionize, Applitools, KaneAI and similar low-code/AI platforms speed up breadth. Coding frameworks (Playwright/Selenium) still pay more for depth — combine both.

## Agentic & MCP testing (emerging)

AI agents that drive Playwright via tools/MCP to explore an app and propose tests are appearing in 2026. Early but worth watching — and great portfolio fodder.

## Prompting for testing

Use AI to brainstorm coverage: "Given this signup form, list boundary and negative test cases" or "generate realistic invalid test data for an email field". You review and curate.

## Risks & limits (say this in interviews)

- **Hallucinated assertions** — AI can write tests that pass but assert the wrong thing. Always review.
- **False confidence** — green AI-generated tests can give a false sense of coverage.
- **Data privacy** — don't paste proprietary code/data into public tools.
- **It can't own quality** — judgment, risk analysis and exploratory testing stay human.
