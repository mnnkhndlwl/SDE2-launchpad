You'll constantly verify that an action in the UI/API produced the right data in the database.

## The core you need

```sql
-- filter
SELECT id, email, status FROM users WHERE status = 'active';

-- sort + limit
SELECT * FROM orders ORDER BY created_at DESC LIMIT 10;

-- aggregate
SELECT status, COUNT(*) FROM orders GROUP BY status;

-- only groups matching a condition
SELECT user_id, COUNT(*) AS n FROM orders
GROUP BY user_id HAVING COUNT(*) > 5;
```

`WHERE` filters rows **before** grouping; `HAVING` filters **after** grouping.

## Joins

```sql
-- INNER: only matching rows in both tables
SELECT u.email, o.total
FROM users u
JOIN orders o ON o.user_id = u.id;

-- LEFT: all users, even those with no orders (NULL totals)
SELECT u.email, o.total
FROM users u
LEFT JOIN orders o ON o.user_id = u.id;
```

## Classic QA queries

```sql
-- find duplicate emails
SELECT email, COUNT(*) FROM users
GROUP BY email HAVING COUNT(*) > 1;

-- verify an order was created after a UI/API action
SELECT * FROM orders
WHERE user_id = 42 ORDER BY created_at DESC LIMIT 1;
```

<div class="callout tip">
  <div class="ct">Test data hygiene</div>
  Use SQL to set up and tear down test data so runs are repeatable and isolated. Never let one test depend on leftovers from another.
