The decisions: which store, how to index, how to keep data correct under concurrency, and how to scale it.

## SQL vs NoSQL

Pick by **access pattern and consistency needs**, not hype.

- **Relational (SQL)** — structured schema, **joins**, and **ACID transactions**. Choose it when data is relational and correctness matters (payments, orders, anything with invariants across rows). Postgres/MySQL scale further than people think.
- **NoSQL** — flexible schema and horizontal scale, optimized for specific patterns:
  - *Document* (MongoDB) — self-contained documents, flexible shape.
  - *Key-value* (DynamoDB, Redis) — massive scale, simple get/put by key.
  - *Wide-column* (Cassandra) — write-heavy, time-series, huge volume.

The honest framing in an interview: "SQL by default for relational data and transactions; NoSQL when I need a specific access pattern at scale, or flexible schema, and can give up joins / strong consistency."

## Indexing

An index is a sorted data structure (usually a **B-tree**) on a column, so the DB can find rows in **O(log n)** instead of scanning every row (O(n)). The cost: indexes **use space and slow down writes** (every insert/update must maintain them). So index the columns you filter/sort/join on — not everything.

A **composite index** on `(a, b)` helps queries filtering on `a`, or `a` then `b`, but not `b` alone (left-prefix rule).

## ACID transactions

A transaction groups operations so they're **A**tomic (all or nothing), **C**onsistent (invariants hold), **I**solated (concurrent transactions don't corrupt each other), **D**urable (committed = survives a crash). You need them when multiple writes must succeed together — transferring money debits one account and credits another; partial completion is a bug.

## The N+1 trap

Fetching a list, then firing one query per item, is **N+1 queries** — a silent performance killer (1 query for 100 posts, then 100 queries for each post's author = 101 round trips). Fix it by **eager loading**: a join, or a single `WHERE author_id IN (...)` batch.

## Scaling: replication vs sharding

- **Replication** — copy data to replicas; serve reads from them. Scales reads and adds redundancy. Tradeoff: replication lag → reads can be slightly stale.
- **Sharding** — split data across machines by a shard key. Scales writes and storage. Tradeoff: cross-shard joins/transactions are hard; a poor key creates hot shards.
