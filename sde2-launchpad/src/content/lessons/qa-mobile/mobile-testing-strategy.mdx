Mobile testing is not "web testing on a small screen." A phone is a sensor-laden, interruptible, battery-constrained, permission-gated device running one of thousands of hardware/OS combinations, often on a flaky network. Your strategy has to account for conditions a desktop browser never faces.

## What is genuinely unique about mobile

- **Fragmentation** — thousands of Android device models, OEM skins (Samsung One UI, Xiaomi MIUI), chipsets, and screen densities; multiple live iOS and Android OS versions. The same build behaves differently across them.
- **Interrupts** — incoming calls, SMS, push notifications, alarms and pop-ups can fire mid-flow. The app must pause and resume without losing state or crashing.
- **Network conditions** — 5G, 4G, flaky 3G, captive Wi-Fi, offline and **airplane mode**. You must test slow networks, network switching mid-request, and graceful offline behaviour with retry.
- **Battery and power** — high battery drain, behaviour in **low-power/battery-saver mode**, and what happens when the OS throttles or kills background work.
- **Permissions** — runtime permissions (camera, location, contacts, notifications, mic). Test granted, denied, "ask every time", and revoked-while-running.
- **Gestures** — tap, double-tap, long-press, swipe, pinch-zoom, multi-touch, edge-swipe back, pull-to-refresh.
- **Orientation** — portrait/landscape rotation must preserve state; split-screen and foldable postures add more.
- **App lifecycle** — background, foreground, process kill by the OS under memory pressure, and cold vs warm start. State must survive being backgrounded and restored.
- **OS versions** — new OS releases change permissions, background limits and APIs; test the current and previous one or two majors.
- **Screen sizes & densities** — small phones to tablets and foldables; layouts must not clip, overlap or break.
- **Low memory** — the OS reclaims memory aggressively; test behaviour when the app is killed in the background and relaunched.

## The device matrix problem

You cannot test every device. The job is to **choose a representative subset** that maximizes real-user coverage.

Pick devices using data, not guesses:

- **Your own analytics** — which devices, OS versions and screen sizes do *your* users actually have? This is the single most important input.
- **Market share** — regional Android/iOS split and popular models in your target geographies.
- **OS version distribution** — cover the OS versions your users run, weighted by share; always include the latest and at least the previous major.
- **Form-factor extremes** — smallest and largest screens, plus a tablet/foldable if relevant.
- **Risk** — known-problematic OEMs (aggressive background killing), low-end devices for performance, devices with hardware your app uses (NFC, biometrics, dual SIM).

A practical matrix: a few **high-priority** devices tested deeply on every build, a **medium tier** of cloud devices in regression, and a **long tail** sampled periodically. Tier by coverage value, not by what's on someone's desk.

## Real devices vs emulators vs cloud farms

| | Strength | Weakness |
|---|---|---|
| **Emulator / simulator** | Free, fast, scriptable, parallel, great in CI | No real sensors, network, battery, OEM skins or true performance |
| **Real devices (in-house)** | True behaviour: sensors, performance, push, biometrics | Costly to buy/maintain; limited models; charging/storage upkeep |
| **Cloud device farms** | Hundreds of real models/OS versions on demand, parallel runs | Cost per minute; some latency; security/data constraints |

Cloud farms — **BrowserStack, Sauce Labs, LambdaTest** (and Firebase Test Lab, AWS Device Farm) — give breadth without owning hardware. The mature strategy: **emulators for fast functional checks in CI, a small in-house real-device set for daily smoke and sensor work, and a cloud farm for broad real-device regression** before release.

## Types of mobile testing

- **Functional** — features work as specified across the device matrix.
- **Performance** — app launch time, screen-render and scroll smoothness (jank/frame drops), responsiveness under load and on low-end hardware.
- **Interrupt** — calls, SMS, notifications, alarms, low-battery warnings interrupting flows; app resumes cleanly.
- **Installation / upgrade** — fresh install, upgrade from previous versions (data migration must not lose user data), reinstall, and downgrade where applicable.
- **Localization & internationalization** — translated strings, right-to-left layouts, date/number/currency formats, text that expands in other languages without clipping.
- **Accessibility** — screen readers (TalkBack on Android, VoiceOver on iOS), dynamic/large font sizes, color contrast, touch-target size, focus order.
- **Security** — secure storage of tokens/PII, certificate pinning, no secrets in logs, safe deep links, jailbreak/root handling, encrypted transport.
- **Battery & memory** — drain over a session, behaviour in battery-saver mode, memory leaks, and behaviour when the OS kills the backgrounded process.
- **Compatibility** — across OS versions, screen sizes/densities, and OEM skins.
- **Network** — slow/flaky networks, offline, network switching, and timeouts/retries.

## App types and how testing shifts

- **Native** — full device-feature access; richest performance and sensor testing; platform-native or Appium automation.
- **Hybrid** — native shell with WebView content; test both the native chrome and the embedded web, including context switching and offline web caching.
- **PWA / mobile web** — runs in the browser (and can be "installed" to the home screen); test like web but on mobile viewports, plus service-worker offline behaviour, add-to-home-screen, and push where supported. No app-store gate.

## Store guidelines & release testing

Shipping to the App Store / Play Store adds checks a website never needs:

- **Store policy compliance** — Apple App Store Review Guidelines and Google Play policies (data-safety disclosures, permissions justification, content rules) — a rejection blocks release.
- **Metadata & assets** — app name, screenshots per device size, icon, privacy labels/data-safety form.
- **Staged rollout** — release to a small percentage first and watch crash-free rate and reviews before going wide.
- **Upgrade path** — verify users on older versions upgrade cleanly with data intact.
- **Crash/ANR monitoring** — track crash-free users and Android ANRs post-release (Crashlytics/Play Vitals); a regression here is a rollback trigger.
- **Forced-update / kill-switch** — verify the app can require a minimum version or disable a broken feature server-side.

  <div class="ct">Watch out</div>
  Mobile releases are not instantly reversible like a web deploy. Once a build is on users' phones you can't pull it back — you can only ship a new version through review and hope they update. That's why staged rollouts, crash monitoring and server-side kill-switches matter so much.

## Interview questions & model answers

**Q1. What makes mobile testing different from web testing?**
Mobile adds conditions web doesn't have: device/OS fragmentation, hardware sensors, runtime permissions, interrupts (calls/notifications), variable and offline networks, battery/power constraints, orientation and the app lifecycle (background/foreground/process-kill). Releases also go through app stores and can't be instantly rolled back. So a mobile strategy tests conditions and state transitions, not just feature happy paths.

**Q2. How would you choose which devices to test on?**
Start from real data — your own analytics on which devices, OS versions and screen sizes your users actually use — then layer in regional market share. I tier them: a few high-value devices tested every build, a medium cloud tier in regression, and a sampled long tail. I deliberately include the latest and previous OS majors, the smallest and largest screens, and any device with hardware my app depends on or known-problematic OEMs.

**Q3. Real devices, emulators or cloud farms — when do you use each?**
Emulators for fast functional checks in CI because they're free and parallel. A small in-house real-device set for daily smoke and anything needing real sensors, performance or push. A cloud farm like BrowserStack or Sauce for broad real-device coverage across many models and OS versions before release. Emulators can't reproduce real performance, sensors, OEM skins or network conditions, so they never fully replace real devices.

**Q4. How would you test that an incoming call doesn't break checkout?**
This is interrupt testing. I'd start a checkout, trigger an incoming call (real call, or `adb` telephony emulation on Android), let it ring/answer, then return to the app and verify the checkout state is intact and no double-charge or crash occurred. I'd repeat for notifications, alarms and the app being backgrounded mid-payment, and verify lifecycle callbacks save and restore state.

**Q5. How do you test offline and poor-network behaviour?**
Use network throttling/airplane mode to simulate offline, slow 3G and flaky connections, and switch networks mid-request (Wi-Fi to cellular). I verify graceful degradation: clear offline messaging, queued actions, automatic retry on reconnect, no data loss, and sane timeouts rather than indefinite spinners. Tools: device airplane mode, OS network conditioner, `adb` shell, and cloud-farm network profiles.

**Q6. What extra testing does an app-store release need that a website doesn't?**
Store-policy compliance (Apple/Google guidelines, data-safety disclosures), correct metadata and per-size screenshots, clean install AND upgrade paths with data migration, staged rollout with crash-free-rate and review monitoring, and a server-side kill-switch / forced-update mechanism. Because you can't instantly roll back a shipped build, post-release crash/ANR monitoring is part of the test plan.

**Q7. Native, hybrid and PWA — how does your testing change?**
Native gives full sensor/performance scope and uses platform-native or Appium automation. Hybrid means also testing the embedded WebView and native/web context switching, plus offline web caching. PWA/mobile-web is tested like web on mobile viewports but adds service-worker offline behaviour, add-to-home-screen and push where supported — and skips the app-store gate entirely.

## Common mistakes

- **Treating mobile as small-screen web** — missing interrupts, lifecycle, permissions and network entirely.
- **Choosing devices by gut** instead of analytics and market share — over-testing flagships, under-testing the low-end and old OS versions where bugs hide.
- **Only testing on emulators** — shipping performance, sensor and OEM-skin bugs real devices would have caught.
- **Ignoring the offline/poor-network path** — apps that hang forever on a spinner in a tunnel.
- **Skipping upgrade testing** — fresh installs pass while an upgrade silently wipes user data.
- **Forgetting permission states** — only testing "allow," never "deny" or "revoked-while-running."
- **No interrupt testing** — calls/notifications mid-flow cause crashes or lost state in production.
- **Treating a mobile release like a web deploy** — assuming instant rollback when you actually need staged rollout and a kill-switch.
- **Neglecting accessibility and localization** until late, when text-expansion clipping and screen-reader gaps are expensive to fix.
