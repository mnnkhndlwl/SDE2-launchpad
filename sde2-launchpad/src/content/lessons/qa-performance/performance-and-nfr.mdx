Functional testing asks "does it work?"; non-functional testing asks "does it work *well enough* — fast, secure, usable for everyone?"

## Performance testing

Types:

- **Load** — expected traffic; does it hold up?
- **Stress** — beyond expected; where does it break?
- **Soak/endurance** — sustained load; memory leaks?
- **Spike** — sudden surges.

Key metrics: **latency** (response time), **throughput** (requests/sec), **error rate**, and percentiles — **p95/p99** (95%/99% of requests are faster than this). Averages lie; percentiles tell the truth about the slow tail.

```js
// k6 — a simple load test
import http from 'k6/http';
import { check } from 'k6';
export const options = { vus: 50, duration: '1m',
  thresholds: { http_req_duration: ['p(95)<500'] } };
export default function () {
  const res = http.get('https://test.k6.io');
  check(res, { 'status 200': r => r.status === 200 });
}
```

**k6** is the modern scriptable choice; **JMeter** is the GUI-driven classic still widely used.

## Security testing (basics every tester should do)

Know the **OWASP Top 10**: injection (SQL/XSS), broken authentication, broken access control (e.g. **IDOR** — changing an id in the URL to see someone else's data), security misconfiguration, sensitive data exposure. Run an automated scan with **OWASP ZAP** and always test that auth and authorization actually block what they should.

## Accessibility (a11y)

Test against **WCAG**: keyboard-only navigation, color contrast, alt text, labels, focus order. Automate the easy 30-40% with **axe-core** (or `@axe-core/playwright`); do the rest manually with a screen reader.
